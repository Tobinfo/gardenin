# gardenin Audit - 2026-05-20

This audit compares the intended product behavior from `docs/DECISIONS.md` and `docs/NEXT_STEPS.md` against the current browser app implementation.

## Executive Summary

The app has moved in the right direction on the hard foundations: real Pl@ntNet calls are server-side, weather is live by ZIP, photos are cropped before durable storage, saved plant photos are visible, and a local saved-plant matcher exists.

The main drift is product-mode confusion. The UI currently lets `ID only`, saved garden recognition, provider fallback, quick add, and care tracking overlap in ways that make the app feel inconsistent. The screenshot shows this clearly: the user is in an `ID only` result state while the right side still shows garden plant tracking and care actions, and the box label says a low-confidence outside-plant candidate even though a saved garden plant is present.

The biggest technical scaling issue is that all crop photos are stored as base64 strings in browser `localStorage`. That is acceptable for a prototype, but it will not scale to real users, cross-device use, reliable training data, or data export/delete compliance.

## Intention Versus Result

### 1. Photo-first garden records

Intention:

- The app should not give plant-specific care guidance without a current photo/observation first.
- Quick/manual entry should attach to an observation photo before it becomes care guidance.

Current result:

- `Quick add` can create a plant record with no current photo.
- `openQuickAdd()` creates a manual candidate and opens the review panel without any camera capture.
- `savePlant()` saves that plant and immediately creates care guidance.

Impact:

- This violates the photo-before-advice rule.
- It creates weak records that cannot train recognition and cannot be audited later.

Recommended fix:

- Split quick add into two modes:
  - `Quick add from camera`: capture/freeze a crop first, then manual species/name.
  - `Draft plant`: allowed without care recommendations until a photo is attached.

### 2. Saved-plant recognition should be repository-first

Intention:

- Known-plant point-and-ID should recognize from the user's own saved crop-photo repository first.
- If local recognition fails, the app can fall back to Pl@ntNet.

Current result:

- The app does check local crop photos first.
- But after Pl@ntNet returns a candidate, `knownPlantFor()` can still auto-trigger saved-plant recognition by matching species name against an existing plant.

Impact:

- A provider result can look like gardenin personally recognized the saved plant.
- That blurs the line between "our repository recognized your plant" and "Pl@ntNet guessed a species you already have."
- The screenshot likely reflects this confusion: a low-confidence provider-style result appears while garden tracking remains visually active.

Recommended fix:

- Only flash/pop up `Recognized Plant Name?` when `metadata.source === "local-repository"`.
- If Pl@ntNet returns a species already in the garden, show a different frozen review state: `This may be one of your saved Red columbine plants`.
- Require explicit confirmation before logging an observation from provider fallback.

### 3. ID-only should stay separate from garden tracking

Intention:

- `ID only` identifies outside/non-garden plants.
- It should not add care tracking or blend into saved garden records.
- Optional save goes to personal ID gallery only.

Current result:

- The ID-only flow does not save a garden plant, which is correct.
- But it reuses the same scan result area, detection box, and page layout as the garden scan.
- The right-side garden plant panel stays dominant during ID-only, and the recognition debug says `ID only` while garden care actions remain visible.

Impact:

- The mode boundary is visually weak.
- Users can read the screen as if the ID-only candidate is related to the saved garden plant list.

Recommended fix:

- Add an explicit mode header/state for the scan panel: `Garden scan` vs `ID only`.
- During ID-only result, visually suppress or de-emphasize garden care actions until the user closes the ID-only result.
- Rename `Confirm ID` to `Done` or `Use this ID` so it cannot be confused with `Add plant`.

### 4. Low-confidence workflow is incomplete

Intention:

- Low confidence should ask for retry and collect more photos without spending extra provider calls immediately.
- The app should choose the best of multiple photos before calling the provider.

Current result:

- ID-only does this: it captures 3 photos, scores them, and sends the best crop.
- Normal garden scan does not yet do this. A low-confidence garden scan shows retry/manual/more-photos options after a provider call.

Impact:

- The credit-saving idea is only implemented for ID-only.
- Main garden scan can still spend a provider call on a weak first frame.

Recommended fix:

- Bring the 3-photo best-shot gate into normal garden scan before the first provider call when local repository matching fails or confidence is low.
- Keep the live camera preview active during those extra captures.

### 5. Training photos are useful but too tightly bound to local browser storage

Intention:

- Confirmed IDs should become a growing personal recognition repository.
- Users should be able to view/export/delete photos.

Current result:

- Per-plant photo library exists and supports export/delete.
- Recognition uses saved crop photos.
- All photos are base64 in `localStorage`.

Impact:

- Browser storage quota will be reached quickly.
- Data disappears on browser reset/profile changes.
- No cross-device continuity.
- No server-side training corpus or account deletion/export workflow.

Recommended fix:

- Move crop photos to object storage as the next major architecture step.
- Keep `localStorage` only as an offline cache/index after hosted persistence exists.

## Current Product Issues

1. `Quick add` violates photo-first care guidance.

   Current location: `prototype/app.js`, `openQuickAdd()` and `savePlant()`.

2. Provider species matches can masquerade as saved-plant recognition.

   Current location: `prototype/app.js`, `showCandidates()` calls `knownPlantFor()` after provider results.

3. ID-only mode is visually blended with garden tracking.

   Current location: shared scan result UI in `prototype/index.html` and `selectIdOnlyCandidate()`.

4. Normal garden scan does not use the 3-photo best-shot flow before provider calls.

   Current location: `captureAndIdentify()` sends the first crop directly to `identifyFromImage()`.

5. Local recognition is a color/texture heuristic, not a robust plant model.

   Current location: `prototype/photo-recognition.js`.

6. Detection boxes are the fixed focus box, not real object segmentation.

   Current location: Pl@ntNet candidates receive `focusBox` as `observationBox`; no true plant boundary is returned.

7. Photos are base64 in `localStorage`.

   Current location: `storageKey`, `idOnlyGalleryKey`, `savePlants()`, and `saveIdOnlyGallery()`.

8. Consent is too coarse for future hosted/product use.

   Current result: yes/no photo use for personal recognition. Missing separate controls for personal recognition, product improvement, deletion/export, and account-level settings.

9. Weather is live but not yet a durable decision input.

   Current result: weather affects recommendations in-session and is copied into plant records at save time, but there are no hosted `weather_snapshots` or historical decision records.

10. Perenual is connected but not productized.

   Current result: provider status can show Perenual key exists, but saved plant care profiles are not enriched from Perenual yet.

11. Render URL/name still says `floraos`.

   Current result: code/docs are moving to `gardenin`, but the public URL remains `https://floraos.onrender.com`.

12. There is no account/user identity.

   Current result: all records are browser-local, so the app cannot support real users, shared devices, sync, or deletion requests yet.

## Scaling Drift Risks

### Data Model Drift

The browser app currently uses nested objects directly in local storage. The API contract defines future hosted models, but the app is not yet using those models through an adapter. Every new browser-only field increases migration work.

Mitigation:

- Add a client-side repository layer now, even before hosted storage.
- Route all plant/photo/care mutations through that layer.

### Photo Storage Drift

Base64 crop strings inside plant records are simple but brittle.

Mitigation:

- Introduce `PlantPhoto` records with stable IDs immediately.
- Store object URLs locally for now, then swap storage backend later.

### Recognition Trust Drift

The app currently has three recognition sources:

- local saved photos
- Pl@ntNet
- manual/quick add

The UI does not always clearly separate them.

Mitigation:

- Make the recognition source first-class in every result:
  - `Recognized from your garden photos`
  - `Identified by Pl@ntNet`
  - `Entered manually`
- Only the first one should use flash-only walking recognition.

### Consent/Compliance Drift

The current crop-only behavior is a good default, but the consent language will not be enough once photos are hosted.

Mitigation:

- Add a data settings screen before hosted persistence:
  - personal recognition on/off
  - product improvement opt-in/out
  - export my data
  - delete my data
  - delete individual photos

### Mobile Drift

The browser workflow is still changing quickly. Native work should not start with the full garden-care flow yet.

Mitigation:

- Keep first mobile proof narrow:
  - camera crop
  - ID-only 3-photo capture
  - best-photo scoring
  - server Pl@ntNet call
  - optional ID gallery save

## Highest Priority Fixes

1. Stop provider results from triggering saved-plant flash recognition.
2. Make ID-only visually and logically separate from garden tracking.
3. Block care-tracked quick/manual plants unless a photo is attached, or mark them as drafts.
4. Bring 3-photo best-shot capture into normal garden scan.
5. Add a client-side data repository layer so hosted persistence can replace local storage without rewriting the UI.

## Follow-Up Fix Pass

Implemented after this audit:

- Provider species matches no longer trigger saved-plant flash recognition. Flash recognition is reserved for local repository matches.
- ID-only now sets an explicit app mode and de-emphasizes garden care actions while active.
- Quick add requires a current plant-box crop before opening the save form.
- Normal Garden Scan now captures three plant-box photos, scores them, and submits only the strongest crop. Extra crops can become training photos after the plant is saved and photo use is allowed.

Still pending:

- Manual camera verification of the new three-photo Garden Scan flow.
- Hosted object storage/database so these crop photos are not stuck in browser `localStorage`.
- A stronger recognition model or embeddings once enough labeled crop photos exist.

## What Is Working

- Hosted `/api/status` reports Pl@ntNet and Perenual keys are present.
- Hosted `/api/mobile-contract` is live.
- `npm.cmd run check` passes.
- Server-side provider boundary exists.
- ZIP weather is live through `/api/weather`.
- Crop-only storage is implemented for current saved records.
- Per-plant photo library exists.
- ID-only captures three photos and submits the best one.
- Local recognition refuses plants with fewer than three photos.
